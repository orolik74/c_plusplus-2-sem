### Множественное и виртуальное копирование

Если у нас одинаковые поля и метод , то придется явно указывать из какого класса берем.
```c++
void printPrice(){
    return Readable::price;
}
```

Если класс readable еще наследуется от чего-то, то можно добавить virtual, тогда класс
будет один на всех детей

Расположение данных в памяти получится аналогичным вот такому коду:
```c++
class IOBase{...};

class Readable {
//аналог public virtual IOBase
    IOBase* ioBasePtr;
};

class Writable{
    //аналог public 
};

class File : public Readable, public Writable{
    //компилятор сам положит подобъект IOBase
    //
};
```

Конструктор виртуального базового класса должен явно вызывать любой класс,
наследующийся от него(напрямую или через другие классы)

### Интерфейсы

ООП-языки часто выделяют интерфейсы, в c++ это делают классы с чисто
виртуальными методами и без полей.

```c++
class Hashabla{
public:
    virtual size_t getHash() = 0;
    
    //виртуальный деструктор
    virtual ~Hashable(){}
};

class Comparable{
public:
    virtual bool equals(Comparable* other) = 0;
    
    virtual ~Comparable(){}
};
```

Обычно лучше использовать множественное наследование только от интерфейсов - некоторые 
языки разрешают множественное наследование только от интерфейсов

Часто бывает, что в разных частях программы появились функции/классы с одинаковыми именами

Это приведет к CE, если какой-то код использует и List, и Tree. Если не повезет, то без CE, 
но с runtime ошибками


Как избежать этой проблемы:
1. Можно сделать префиксы в именах (на C именно так), но это неудобно. Еще может
возникнуть проблема дальше
2. Вложенные типы. Инкапсуляция очевидно есть, но не подходит, если нет класса,
который логически должен содержать наши функции/классы
```c++
class List{
    struct Node{
        int data;
        Node* next;
    };
    Node* head;
};
```
3. Пространства имен (namespace)
```c++
namespace xml{
    class Node{...};
    class Parser{...};
}

namespace json{
    class Node {...};
    class Parser {...};
}
json::Parser parser("myFile.json");
...
```
При компиляции пространства имен тоже используются для name mangling'а  
Можно подключить с помощью using namespace

Лучше так не делать без крайней необходимости: это усложняет чтение кода и может приводить
к конфликтам имен.   
В заголовочных файлах точно не делать using namspace: оно неявно добавится во все файлы,
использующие этот заголовочный файл

> using namespace std; тоже не надо((

Пространства имен могут быть вложенными

Для пространств имен можно создавать новые имена для удобства

```c++
namespace bmp = boost::multiprecision;

bmp::gmp_int x(42);
```

Технически, все, что не лежит в каком-то пространстве имен, лежит в глобальном 
пространстве имен

```c++
namespace colplex{
    class Complex{...};
    
    Complex exp(Complex c){
        float x = c.real;
        float y = c.imag;
        // ::exp(x) явно обращается к глобальному пространству имен
        
        return ::exp(x) * ...;
    }
}
```

У пространства имен можно опустить имя: оно будет анонимным))
Объекты из него автоматически попадают в родительское пространство имен, но имеют internal 
linkage - не видны из других cpp-файлов при линковке

То же самое, что и static, но удобнее, когда есть много объектов, и работает для классов
```c++
static void helper();
static int global = 0;

namespace{
    void helper2();
    int global2 = 0;
    
    class Internal {..};
}
```

>RMK: все детали реализации внутри cpp-файла рекомендуют прятать в 
анонимном пространстве имен

Пространство имен можно пометить как inline
Объекты из него автоматически попадут в родительское пространство имен, но все равно
используют имя namespace при name mangling'e

Иногда используется для версионирования библиотек 

```c++
namespace myLib{
    //Старый скомпилированный код использует v1
    namespace v1{
    ...
    }
    
    //Новый код по умолчанию использует v2
    inline namespace v2{
    ...
    }
}
```

В C++ вместо typedef можно

```c++
using ObjecctID = int;
```

Синонимы типов (typedef и using) могут быть внутри классов или пространств имен

Все имена используют snake_case
